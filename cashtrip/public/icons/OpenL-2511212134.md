# VI. Observações e Mapeamento Crítico (Solução para Inconsistência) 

Este é o ponto crucial para resolver o seu problema na plataforma Antigravity/MCP. Muitas plataformas de Design System (como o Antigravity) esperam um mapeamento de cor 

semântico (e.g., primary é sempre primary, success é sempre success). 

A. Inconsistência Resolvida na Cor de Sucesso: 

● Ação: A cor Status-Success foi padronizada para o verde (#10B981) em ambos os temas (Claro e Escuro). Isso garante que o feedback de sucesso seja universalmente reconhecido na aplicação. 

● Mapeamento: A cor laranja/pêssego (#FF896F), que estava sendo usada para "Success" no tema escuro original, foi reclassificada como Status-Filter para manter o uso nos seus filtros e evitar erros de interpretação. 

B. Uso de Opacidade (Transparência): 

As cores que usam porcentagem de opacidade (e.g., #FFFFFF + 60%) são frequentemente a fonte de erros na conversão de Design System para plataformas de desenvolvimento, pois a plataforma pode interpretar a opacidade de forma diferente dependendo do modo de cor (RGB vs RGBA). 

● Recomendação: O ideal é calcular o valor HEX final com a opacidade aplicada sobre o fundo alvo e usar o código HEX/RGB completo com transparência (RGBA) , se possível (e.g., #FFFFFF + 60% sobre um fundo branco é diferente do mesmo valor sobre um fundo escuro). 

○ Exemplo: Se o backgroundReturn no tema escuro é 40% de opacidade sobre #64748B, a variável no MCP deve ser o resultado final da cor (ou você deve garantir que o sistema de design do MCP/Antigravity suporte a variável $bg-return-opacity: rgba(100, 116, 139, 0.4);).
